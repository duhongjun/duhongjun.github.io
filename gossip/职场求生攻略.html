# 职业素养篇

# 如何安排工作优先级

工作中那么多事情，我要如何安排优先级？

# 基于工作本身的性质，把工作划分为

  1. 公司发展计划(公司发展重点方向相关的工作)
  2. 安全相关的事情
  3. 生产上(部署,交付等)的事情

# 从合作角度

  • 根据经理安排的优先级来做。
  • 临时安排的事情, 有时候更急一些，可能也不用耗费很长时间。所以这种事情要先做。
  • 如果一个事情需要别的部门配合，那么优先做。
  • 如果事情没有太明显的轻重缓急, 优先做那些阻塞了别人工作的事情。有些事情来自组内，有些来自组间合作。良好的合作关系就是这样一点点打磨出来的。

# 基于事情本身

给工作排好优先级之后，我们还要注意工作内部各项事情的优先级。工作需要拆成不同的步骤来实施，这些步骤也有优先级，基本道理和上面讲的都一样。

我们做事情的时候，如果能把其中的每一步都想清楚，理清依赖关系，安排得井井有条，就能事半功倍。

# 使用邮件

“邮件不是用来沟通的，它是用来撕扯和甩锅的。” 😶

# 邮件的特性

  1. 异步交流：邮件是一种异步交流的方式，双方有足够的时间准备邮件内容。
  2. 无法修改：邮件内容无法修改，这是邮件可靠的基石。
  3. 方便扩散：邮件有邮件组，可以很方便地把相关人员加进来，并且保留邮件历史记录。

# 邮件是公司内部的合同

  • 邮件的"确认功能" -> 设计确认, 通知相关人员
  • 邮件的“证据链”功能 -> 定下的事情, 可追溯
  • 邮件的“沟通协调”功能 -> 大促,各组间的沟通
  • 邮件的“防遗忘”功能 -> 新业务接入
  • 邮件的“广而告之”功能 -> 技术升级和 Bug 修复

# 发送邮件的小技巧

看清收件人，再决定说什么。

学会抄送老板。如果想催促进度，就抄上对方的老板。如果觉得自己搞不定，记得抄上自己的老板。

用好邮件组，该加人的时候加人，做到应通知, 尽通知。

要细心, 记得多检查一下邮件内容和标题, 不要有写错内容或者有错别字。


# 多交流

# 为什么程序员普遍不喜欢交流？

  • 工作被打断严重影响效率, 所有的程序员，都非常讨厌自己的工作被打断。
  • 交流不能直接帮程序员完成工作, 会议结束之后，程序员的工作才算真正开始, 程序员其实只是交流中的“利益受损者”，我们排斥的不是交流本身，而是厌烦交流中时间的“浪费”。

# 爱上交流的第一步：正视和人的交流

在做事情的时候，优秀的交流能力是必须的。如果你能够展示自己在交流能力上的优势，组内外的同事也更愿意和你合作，经理也会优先考虑让你承担更多的责任，对自己的发展有很大的好处。 交流的好处可以从两个方面来说，一个是输入，一个是输出。

# 输入

  1. 获取更多的信息；
  2. 理解公司的业务；
  3. 加深对行业的理解；
  4. 发现新的机会。

# 输出

  1. 赢得自己的声誉；
  2. 树立自己的影响力；
  3. 赢得同事和经理的信任，承担更大的责任。

# 程序员交流的技巧

  • 换位思考，注意受众
  • 交流要带有足够的信息
  • 先说重点和结论

# 发挥主观能动性

# 为什么要发挥主观能动性

  • 软件研发的复杂性，导致没有标准可以参考，也导致很多事情无法在设计之初预见。
  • 现代软件研发快速迭代，导致需求和需要应对的问题也在快速变化，程序员不能固守之前的设计。
  • 做完不等于做好，能用不等于好用。

# 如何发挥主观能动性

  • 交付思维：站在用户的角度，交付真正解决用户问题的软件系统。
  • 注意时间： 发挥主观能动性的一个代价，就是会用掉更多的时间。比起功能的完美，在规定的时间内实现基本功能，才是优先级更高的事情。

# 责任的边界

程序员的职责范围仅仅只是被安排的任务吗？程序员需要对哪些事情负责?

# 对自己的基本能力负责

  • 持续精进技术能力。
  • 熟悉公司的内部系统

# 对安排的工作负责

  • 尽自己的努力完成工作。
  • 发现自己无法完成工作时，要尽早告诉经理。

如果有额外的付出，可以将自己做的计划外的事情说出来，这样一方面让大家认可你做的工作，另一方面，也是能够让大家对这类事情的工作量估计更合理。

# 对工作时间负责

  • 如果是弹性工作制，尽量在公司大多数人上上班前到，保证别人找你时在线。
  • 准时参加会议，如果有事不能参加，要及时和大家同步。

# 让别人看到自己的工作

大小事都报备不合适，那么不报备显得自己什么都没做，也憋屈。可以这样，小事也是跟大事相关的，这些琐碎的、浪费时间的事情，为什么不得不处理呢？就是因为跟这些事情相关的更高层级的事情更重要。把自己做的事情跟这些重要的事情关联起来，就显出自己做的事情的分量了。

举个例子。

比如说是给业务增加新的API和数据表。如果只是单纯的说

“增加了一个表，添加了几个API操作表里的数据”

这样谁看着都枯燥。也突出不了自己的做的贡献。从技术角度看，也就是CURD，没啥好说的。

如果你在做API和数据库表的时候，多跟业务方聊两句，问问这个API是给哪个项目的，是干什么用的等等，那你报备的时候就可以这么些：

“为公司重点项目购物车二期新增三个API，一个购物车数据表，提供了购物车二期项目中对Android和iOS应用中购物车操作的支持。”

所以，觉得做的事情没意思的时候，就问问业务方，因为我们做的东西的意义都是业务提供的。正如你说的那样，注意沟通，沟通让你获取更多的信息，see the big picture。


# 职业选择篇

# 怎么才叫“跟对人”

“跟对人”的表现：

  • 认同彼此的价值观和人格 -> 有担当 / 优秀、真诚、公平
  • 定期和你交流 -> 指出问题，表扬有点，给出发展建议
  • 可以互相讨论甚至争论 -> 发挥主观能动性的前提，能够得到更好、更完整的结论。
  • 资源和利益的分配 -> 给手下人新的机会， 给手下人发展的时间，争取利益、并能公平分配。

# 跟对人为什么比做对事更重要？

  • 奇葩的经理是存在的。
  • 别对公司抱有太多的幻想 -> 你被替换的代价并没有你想的那么高。
  • 跟对人，才能做对事 -> 跟错人，做事阻力大，观念不合，即使做成了利益也不会分配到自己这里。

# 跳槽之前你想清楚这些事情了吗?

跳槽前的chekcList:

  1. 为什么想跳槽?
  • 心态 -> 是自己的问题还是外部的问题? 换公司无法解决自身的问题
  • 主动求变 -> 新行业、新方向 / 自己的兴趣
  • 薪资待遇 -> 不要让薪资成为你跳槽的唯一原因； 公司：稳定 + 合理
  1. 为什么不应该频繁跳槽
  • 公司不愿意聘用和培养频繁跳槽的人
  • 长期来看，在一个公司的积累和成长最值钱 (行业深耕)
  • 不一定非要跳槽，可以考虑内部转岗

# 准备简历和面试

面试的逻辑：与面试官合作，展示自己的优势和优秀之处。

  1. 写简历
  • 精简工作经历 -> 解决的问题、自己的成长
  • 经历倒叙
  • 突出重点，强调自己的优势
  • 排版工整
  1. 准备面试
  • 针对公司面试风格，提前做准备。
  • 为什么换工作、刷算法和数据结构题/ 弄清自己的优势、突出工作成果、梳理自己的技术，准备面试、建议内推
  1. 面试中需要注意什么？
  • 不要勉强
  • 注意反馈，沟通流畅，表达清晰
  • 不推荐靠面经面试
  • 其他：准时到，不要和面试官较劲

# 职业规划：转管理是程序员的终极选择吗？

转管理后的不同： 管理岗要通过让手下创造更多地价值，来实现自己的价值。

一线经理要做什么？

  • 对内：
  1. 从个人视角转向管理视角，发挥团队的最大战斗力。
  2. 管理绩效，合理激励。
  • 对外：
  1. 计划和安排，争取资源。
  2. 进行各种交流，做出承诺和决策。
  • 对未来：
  1. 理解公司发展方向。
  2. 培养人才，发展团队。
  3. 获取客户，赢得认可。

什么样的人适合转管理？

  • 喜欢和人打交道
  • 会经营，有眼光，有干劲
  • 能够承受压力
  • 有远大抱负

转不转管理呢？

  • 管理的临界级别确实比个人岗要高一到两个级别
  • 无论是不是转管理，临界级别在那里，成长都是有瓶颈的。
  • 程序要要有专攻的方向，比如业务，比如架构。
  • 无论是否转管理，越成长，越需要交流，获取信息，整合信息。

# 职业规划 ：程序员加入创业公司，这些事情你想清楚了吗？

想要获得超额回报，最根本的逻辑呢，是要成为公司核心竞争力的一部分，要能够在公司创造价值的链条上成为重要的、不可替代的一环。仅仅是重要还不行，如果很容易被替代的话，能分到手的利益很可能也只是象征性的。

创业，也许大概率不会有金钱上的收获，但是可以让你看清自己的潜力。

  1. 认清创业公司
  • 做项目变现的公司“不是”。 如外包外派等。
  • 小公司“不一定是”
  • 创造新事物的公司才”是“
  1. 加入创业公司，需要权衡哪些事？
  • 提前算好收支这笔账
  • 选择公司就是选择团队
  • 认可创始人
  • 亲兄弟，明算账
  • 准备好强大的精神和肉体
  • 能持续跟得上公司的发展
  1. 创业 V.S. 普通工作
  • 创业是地狱难度，工作是普通难度
  • 其实创业是一种心态，在普通工作中，如果用创业的心态工作，依然可以收获更多的回报
  • 创业即使失败，也可以提前认清自己，收获一段难忘的回忆。

# 职场情商篇

# 升职：看着周围的人都升职了，我什么时候才能升职？

1. 升职的逻辑

  • 先到下个级别的能力，公司才会考虑给升职
  • 稳定输出工作成果，而不是时而做的好，时而做的不行
  • 技术上不仅要绝活，更要能全面掌握工作需要的技术
  • 资历是升职必须要考虑的因素，也是对老员工各种积累和软能力的认可

2. 程序员的三个阶段

  • 写出好代码，代码不再是问题
  • 稳稳地搞定一个系统，清楚系统的方方面面，各种细节，系统能力的边界等
  • 了解系统所解决的问题领域，可以让系统发展到一个更高的层面

3. 升职的经理视角

  • 符合升职逻辑
  • 和经理彼此不对付的人，大概率不会得到升职
  • 离职可能性高的人，升职的概率会降低
1. 在公司多做分享，不管有没有人听，
2. 多画图
3. 体系化思考，总结的能力，形成自己的方法论
4. 要有自己落地的action，有case支撑。

# 职场政治：我只想好好干活，职场政治和我有什么关系？

1.为什么一定会有职场政治

  • 人少总是小而美，利益分配很简单
  • 人多就会有”江湖“，职场政治就是帮忙搞定职场礼仪分配的

2.一般公司里的那些职场政治

  • 改组：必须要打破老的利益格局，发展才能有突破
  • 合作要考虑利益，要互惠互利，不要轻易动对方的蛋糕

3.程序员应该做什么

  • 尊重职场政治
  • 懂得组织的利益所在

# 面对公司自上而下的技术更新，我该怎么办？

自上而下所推动的技术更新，肯定是伴随利益变动的

程序员如何应对？

  • 了解变化的原因和目的，变化都是取舍的艺术
  • 看清和自己相关技术的发展和趋势，做好应对措施
  • 必须积极应对：或全力以赴履行自己的职责，或明哲保身换个地方

# 沟通技巧：如何跟自己的同事请教问题？

1.顺畅沟通的核心：尊重

2.输出式沟通

  • 尊重被输出人的自主意愿
  • 不要打断别人解决问题的过程
  • 是在帮助别人，不是在显摆自己

3.请教式沟通

  • 没人有义务帮你，所以请尊重别人的时间和精力
  • 自己能搞定的问题不要麻烦别人
  • 探索式的问题更受人欢迎

4.请教式沟通小技巧

  • 不要问相同的问题
  • 请教后积极总结归纳，如果收获很多，可以总结出文档
  • 需要大量帮助时，可以向经理申请专门的人帮助自己

5.向上沟通

  • 先准备好数据，要言之有物
  • 汇报工作时，先说成果和问题
  • 申请资源时，要给出详尽的数据和前因后果
  • 请求帮忙做决策时，要带着解决方案来，不要只带着问题来

# 技术成长篇

# 学习观：程序员如何定义自己的技术舒适区？

1.程序员的技术舒适区，就是自己的技术领地

2.为什么程序员必须一直学习

  • 技术要么发展，要么被淘汰
  • 学到手的技术会一直贬值，不学习，自己就跟着技术一起贬值

3.技术骨架的构成

  • 核心架构设计：这个技术有哪些核心的架构设计
  • 功能模型： 这个技术有哪些功能，功能的接口是什么
  • 状态模型：系统在运行时有哪些状态，状态之间的变化原因是什么
  • 数据模型：这个技术是怎么组织数据的，是怎么处理数据的
  • 线程模型：这个技术有哪些线程，分别是做什么的

4.程序员的舒适状态

  • 在工作中寻找能提供正反馈的学习目标，工作中有相当比例的时间在学习
  • 关注行业发展，了解新技术的基本情况，但是不主动出击去深入学习用不到的技术
  • 没事不找事，遇事不怕事，工作中遇到任何技术问题，都抽时间搞定它
  • 深入学习新技术时，搭建技术骨架，用新知识点缀自己的技术骨架

# 技术观：做程序员，技术观为何如此重要？

1.软件工程师是要用技术把事情做成

2.吃透技术

  • 熟练使用：熟练使用技术解决问题，深谙技术的各种坑
  • 精准掌握：明晰技术的能力边界，准确判断技术能否以及如何满足某个具体的业务场景
  • 知根知底：知道技术的底层实现和关键细节，如果让自己做，可以大致做出一个来

3.需求至上

  • 技术的目的是满足需求
  • 技术的价值是支撑业务
  • 技术要解决用户的问题，而不是证明技术本身更高明

# 技术观：程序员在技术的成长之路上，有哪些陷阱？

1.技术不扎实，不思进取

  • 我们都是从小白做起

# 系统架构：如何从写代码的程序员，成长为软件系统架构师？

1.软件架构师就是构建一个世界，让业务可以在这个世界里运转

2.软件系统架构师做什么

  • 理解要解决的问题
  • 摸清解决问题能使用的资源
  • 定义解决问题的模型
  • 给出模型各个模块的详细设计

3.如何一步步成为架构师

  • 每个程序员都或多或少的在做架构师的事情
  • 夯实技术能力
  • 主动学习架构，注重自己架构师能力的培养
  • 保持开放的心态，不能固守己见

4.架构师的核心能力

  • 来自实践和技术的积累
  • 对行业和技术的产生自己的理解，形成自己的眼界

# 系统集成：为什么最容易出问题的是系统集成？

1.系统集成为什么最难

  • 实际情况可能没有预料到，也可能发生了变化
  • 系统集成意味着有交互，所有没落实的细节问题都会在这暴露出来
  • 系统集成要产出成果，要得到别人的认可

2.软件系统集成的常见问题

  • 内部子模块之间的架构不统一
  • 对外部系统的假设不成立
  • 外部系统配置问题
  • 用户输入数据问题
  • 外部系统传输的数据问题

3.软件系统集成的应对方法

  • 架构要统一
  • 不做假设，尽早验证外部依赖
  • 在系统集成的边界，充分的记录log

# 沟通原则：什么时候应该妥协，什么时候应该坚持？

1.处理冲突的双方要对等

  • 自己说了不算的事情，不要做出承诺
  • 很多和利益相关的冲突，我们程序员只是打辅助，不需要主动参与

2.程序员应该如何应对冲突

  • 找到大家的共识
  • 以共识为基础，把事情向前推进
  • 做事情的方式方法可以妥协
  • 目标必须坚持

3.程序员工作中的那些共识

  • 代码偏好：eslint等业界标准
  • 接口设计：接口设计应该包容
  • 系统架构：大家对问题的理解要统一
  • 新功能：理解新需求，升级架构，以支持新功能

4.冲突的价值

  • 相互切磋，不打不相识
  • 逼迫自己深入思考自己的观点
  • 磨合团队，达成默契

5.程序员一定不要逃避冲突，逃避冲突只能让问题对自己来说更麻烦

# 加班：加班逃不过，如何用正确姿势加班？

1.任务太多

  • 分析是否当前的效率低下
  • 提升效率：升级系统，开发自动化工具
  • 不要习惯低效率的工作，要发挥自己的才能

2.出现了紧急的事情

  • 避免线上问题；增强监控，规范线上流程
  • 紧急的项目：如果收获可观，可以考虑适度阶段性的付出时间

3.公司强制加班

  • 学习和探索新技术
  • 修复大家都在抱怨的小痛点
  • 去遗留系统里淘金，理解系统处理的业务
  • 准备好为系统升级

4.加班的时间，也是我们自己的时间。要利用加班的时间做能提升自己的事情，不要浪费自己的时间

# 焦虑：程序员怎样才能越干越给力？

1.如何应对焦虑感

  • 习惯焦虑，让焦虑督促自己持续学习和进步
  • 找到适合自己的发展道路

2.前期职业？后期职业？

  • 任何职业的从业者都很多，但能脱颖而出的都是少数
  • 任何职业都要努力，选择不能代替努力

  • 软件公司的危机感：比尔盖茨说：微软离倒闭永远只有18个月
  • 软件公司的危机感必须传递给自己的员工，对员工的要求也就更高
  • 程序员必须有创新的精神，有斗志，不安于现状

4.不同类型的职业

  • 产出成果的过程复杂，人才无法批量培养，需要经验积累和发挥自己的主观能动性

5.程序员如何应对

  • 持续技术学习
  • 持续思考业务
  • 吃掉更多的资源，积累自己的经验

# 数据观：在你眼里，数据到底是什么？

1.做系统，数据比软件更重要

  • 数据是软件系统的根本，就好像经济活动里的钱和帐，抓住数据就能看清根本
  • 数据升级和迁移，相比软件升级要难的多，风险也大得多，对于系统的数据架构设计，值的多花点心思
  • 数据被删了，会损失惨重，程序代码被删了，只要不造成数据问题，基本没什么长期影响
  • 数据是新时代的土地，什么都要靠它，而且很难买到，自己积累则要费力更费时

2.与人沟通，没数据，莫开口

  • 先说数据，再说自己的分析，没有数据支撑，大家很难达成共识或得出结论
  • 注明使用的数据的出处，方便你从数据源验证更多的数据
  • 如果有可能，交叉验证，从不同的角度获取相同意义的数据，进行对比和验证
